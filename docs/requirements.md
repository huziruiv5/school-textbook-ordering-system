# 需求分析文档

## 1. 系统概述

学校教材订购系统旨在实现教材订购的数字化管理，系统可细化为两个子系统：**销售系统**和**采购系统**。

### 1.1 销售系统工作过程
1. 教师或学生提交购书单
2. 教材发行人员审核购书单的有效性
3. 审核通过后开发票，登记并返回领书单
4. 教师或学生凭领书单去书库领书

### 1.2 采购系统工作过程
1. 若教材脱销，登记缺书
2. 发缺书单给书库采购人员
3. 新书入库后，发进书通知给教材发行人员

## 2. 外部实体

| 编号 | 外部实体 | 说明 |
|------|----------|------|
| E1 | 教师 (Teacher) | 提交购书单、领取教材 |
| E2 | 学生 (Student) | 提交购书单、领取教材 |
| E3 | 教材发行人员 (Distributor) | 审核购书单、开发票、登记缺书 |
| E4 | 采购人员 (Purchaser) | 采购教材、入库管理 |

## 3. 数据存储

| 编号 | 存储名称 | 说明 |
|------|----------|------|
| D1 | 购书表 (orders) | 存储购书单信息 |
| D2 | 库存表 (inventory) | 存储教材库存信息 |
| D3 | 缺书登记表 (out_of_stock_records) | 存储缺书记录 |
| D4 | 待购教材表 (pending_purchases) | 存储待采购教材信息 |
| D5 | 进/出库表 (inbound_records / outbound_records) | 存储出入库记录 |

辅助数据存储：用户表 (users)、教材信息表 (books)、购书单明细表 (order_items)、发票表 (invoices)

## 6. 系统用例

### 用例1：提交购书单
- **参与者**：教师、学生
- **前置条件**：用户已登录
- **基本流程**：
  1. 用户选择需要购买的教材
  2. 输入购买数量
  3. 提交购书单
  4. 系统验证数据合法性
  5. 保存购书单，状态为"待审核"
- **后置条件**：购书单提交成功，等待审核

### 用例2：审核购书单
- **参与者**：教材发行人员
- **前置条件**：存在待审核的购书单
- **基本流程**：
  1. 查看购书单详情
  2. 审核购书单内容
  3. 选择通过或驳回
  4. 系统更新购书单状态
- **后置条件**：购书单状态更新为"已通过"或"已驳回"

### 用例3：采购教材
- **参与者**：采购人员
- **前置条件**：存在缺书记录或采购需求
- **基本流程**：
  1. 查看缺书记录
  2. 创建采购单
  3. 执行采购
  4. 到货后入库
  5. 系统更新库存和采购单状态
- **后置条件**：库存更新，采购单状态更新为"已到货"

## 7. 非功能需求

- **可用性**：用户界面简洁明了，操作流程清晰
- **数据一致性**：出入库操作需保证库存数据的事务一致性
- **安全性**：基于角色的权限控制，防止越权操作
- **可维护性**：采用三层架构，各层职责清晰
- **响应时间**：页面响应时间不超过3秒
